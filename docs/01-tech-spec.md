# 01 — Техническое задание (ТЗ)

## Контекст
Есть рабочий VPN-сервер `87.121.47.233` (Ubuntu 24.04, Армения), который уже выдаёт
ключи. Нужен Telegram-бот, автоматизирующий полный цикл: приветствие/регистрация,
продажа и продление подписки (ручная оплата на карту + подтверждение админом),
выдача ключей, поддержка, реферальная программа, контроль срока подписки и учёт трафика.

Клиенты подключаются по подписочной ссылке в приложении **Happ**. Ключ формально
показывает ~13 стран, фактически весь трафик идёт через один сервер.

## Что уже есть на сервере (проверено по SSH)
- `xray` (VLESS/REALITY, :443 + инбаунды), `hysteria2` (UDP) — работают.
- `buklproxy` VPS-agent (Node/Express, `:3000`) — выдаёт ключи, авто-экспирация раз в 6ч
  с колбэком на `EXPIRE_CALLBACK_URL`. Подписка на `:2096/sub/<token>`.
- В xray включены per-user stats (`StatsService` на `127.0.0.1:10085`) → трафик по ключу снимаем.

Полный контракт API агента — см. `04-data-contract.md` §2.

## Принятые решения
| Тема | Решение |
|---|---|
| Стек бота | Python 3.12 + aiogram 3 (FSM, reply-клавиатуры, async) |
| База данных | Supabase (Postgres), схема/миграции через Supabase MCP, доступ из бота через asyncpg |
| Хостинг | тот же VPS `87.121.47.233`; polling Telegram + локальный aiohttp для колбэка |
| Оплата | ручная (карта из БД) + подтверждение платёжным админом (MVP) |
| Тексты кнопок/сообщений | в конфиге `texts.py` |
| Динамические значения | тарифы, реквизиты, юзернейм поддержки, id админа, условия рефералки — в БД |
| Ключ пользователя | `telegram_id BIGINT` везде, без UUID |

## Функциональные требования
1. **/start и регистрация** — приветствие новичка + reply-меню; повторному «рады снова видеть».
   Парсинг реф-параметра `?start=<referrer_telegram_id>`.
2. **Главное меню (reply-клавиатура):** Купить/Продлить, Поддержка, Хочу заработать, Мои ключи.
3. **Поддержка** — выдать `support_username` из `settings`.
4. **Мои ключи** — если подписка `active` → показать `sub_url`; иначе «нет активной подписки».
5. **Купить/Продлить** — тарифы рендерятся из БД в reply-кнопки; экран оплаты с суммой и
   реквизитами из БД; «Я оплатил» → ввод ФИО → создание заявки; «Отменить» на всех шагах.
6. **Подтверждение оплаты** — платёжному админу инлайн «Подтвердить/Отклонить»; логика
   добросовестный/недобросовестный (см. `02-business-logic.md`).
7. **Реферальная программа** — реф-ссылка, прогресс, кнопка «Условия»; засчёт при approve.
8. **Контроль срока** — колбэк экспирации от агента + напоминания APScheduler за 24ч.
9. **Учёт трафика** — `user_traffic` по `telegram_id`, сбор джобом из xray stats.

Подробные сценарии и крайние случаи — в `02-business-logic.md`.
Схема данных, enum, env, контракт агента — в `04-data-contract.md`.
Зависимости модулей и риски конфликтов — в `03-modules-and-conflicts.md`.

## Нефункциональные требования
- Идемпотентность вызовов агента (create уже идемпотентен).
- Все секреты в `.env`, не коммитятся. `VPS_AGENT_TOKEN` общий с агентом.
- Webhook-приёмник колбэка слушает только `127.0.0.1`, проверяет Bearer.
- Логирование действий админа в `payments` (`admin_id`, `processed_at`).

## Verification (e2e) — как проверять при реализации
- `vps_agent.health()` → `{ok, xray:true}`.
- /start новым аккаунтом → запись в `users` + приветствие + меню.
- Покупка (добросовестный) → ключ в «Мои ключи», заявка `pending` → Подтвердить →
  `/vps/keys/status` `present:true`.
- Отклонение → `disable`, юзер в `fraud_list`, статус `expired`.
- Реферал: новый по ссылке покупает → после approve `confirmed_count+1`; на 2-м → +30 дн, сброс.
- Экспирация: выставить `expiresAt` в прошлое → агент шлёт `/agent/expire` → статус `expired`.
- Трафик: джоб опросил xray → `user_traffic.total_bytes` вырос.
