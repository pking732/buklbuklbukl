# CLAUDE.md — buklproxy VPN bot (жёсткие правила)

Проект: Telegram-бот продажи VPN-подписок. **Сейчас существует ТОЛЬКО документация, кода нет.**

## Старт каждого контекстного окна (обязательно)
1. Прочитать `plan/file-mapping.md`, затем нужные `docs/`.
2. Опираться на `plan/implementation-plan.md` (изначальный план, порядок этапов).
3. `docs/04-data-contract.md` — single source of truth. При работе с данными открывать ВСЕГДА.

## Перед ЛЮБОЙ правкой — чек (если хоть один ответ «да» → СТОП и спросить пользователя)
- Будет ли конфликт/расхождение с контрактом `docs/04-data-contract.md`?
- Ломается ли текущая модель данных / уже написанный модуль?
- Расходится ли с бизнес-логикой `docs/02-business-logic.md`?
- Что-то отвалится сейчас или станет нереализуемым в будущем?

Если все «нет» — работать. Если «да» — **остановиться и сообщить пользователю**, не чинить молча.

## Железные инварианты (не нарушать)
- Ключ пользователя везде `telegram_id BIGINT`. Никаких внутренних UUID/user_id.
- `device_key_id = "tg"+telegram_id` — только через единый helper, не собирать строку руками.
- Статус подписки меняет ТОЛЬКО `services/subscriptions.py`. Хендлеры не пишут статус напрямую.
- Тексты кнопок/сообщений — в `texts.py`. Тарифы/реквизиты/поддержка/условия — в БД (`settings`/`tariffs`).
- Контракт API агента и `VPS_AGENT_TOKEN` не менять без правки сервера `/opt/vps-agent/`.
- Изменение контракта = правка `docs/04-data-contract.md` + согласование, потом код.
- callback_data админа: `pay:approve:<id>` / `pay:reject:<id>`.
- Все даты — UTC `TIMESTAMPTZ`.

## Параллельная работа агентов
- Один агент = один модуль/слой из `docs/03-modules-and-conflicts.md`. В чужой модуль не лезть.
- Саб-агентам давать `plan/file-mapping.md` и (если нужен сервер) креды из memory `server-access.md`.
